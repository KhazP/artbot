## Repository Front Page

### Code tab

The landing hierarchy should stay:

1. `README.md` as the primary entry point
2. `docs/` for deeper operational and architectural material
3. `CONTRIBUTING.md`, `SUPPORT.md`, and `SECURITY.md` linked from the README

### Issues tab

Keep issue creation narrow:

- bug reports
- feature requests
- no blank issues
- redirect questions to Discussions

Recommended default labels:

- `bug`
- `enhancement`
- `docs`
- `ci`
- `dependencies`
- `security`
- `adapter`
- `sources`
- `good first issue`
- `help wanted`
- `skip-changelog`

Recommended milestones:

- `v0.3.x stabilization`
- `v0.4.0 adapters`
- `v0.4.0 docs and onboarding`

### Pull requests tab

Keep the PR template focused on:

- summary
- linked issue
- change type
- validation
- documentation impact
- secret check

### Actions tab

Use the existing `CI` workflow plus:

- Dependabot updates for `pnpm`
- Dependabot updates for GitHub Actions
- autogenerated release notes using `.github/release.yml`

## Settings Pages

### Settings > General

Recommended feature state:

| Setting                  | Recommended state                                                          |
| ------------------------ | -------------------------------------------------------------------------- |
| Issues                   | On                                                                         |
| Discussions              | On                                                                         |
| Projects                 | Off initially unless you will actively triage there                        |
| Wiki                     | Off initially because the repository already has a `docs/` source of truth |
| Sponsorships             | Off unless you enable GitHub Sponsors                                      |
| Preserve this repository | Optional                                                                   |

Recommended pull request merge settings:

| Setting                            | Recommended state |
| ---------------------------------- | ----------------- |
| Allow merge commits                | Off               |
| Allow squash merging               | On                |
| Allow rebase merging               | On                |
| Automatically delete head branches | On                |

Release readiness item:

- make the repository public only after the first pass through the Security and Rules sections below

### Settings > Moderation

For a public repository, limit review approval power to people with explicit repository access if that option is available in your plan and UI. That prevents drive-by approvals from counting as merge approvals.

### Settings > Rules

Protect `main` with a branch protection rule or ruleset:

| Rule                                   | Recommended state |
| -------------------------------------- | ----------------- |
| Require a pull request before merging  | On                |
| Require approvals                      | `1` minimum       |
| Dismiss stale approvals on new commits | On                |
| Require status checks                  | On                |
| Required status check                  | `CI / verify`     |
| Require conversation resolution        | On                |
| Block force pushes                     | On                |
| Block deletions                        | On                |

If you are still shipping solo and need flexibility, admins can keep bypass permissions temporarily. Remove that bypass once the release process stabilizes.

### Settings > Actions

Recommended repository Actions settings:

| Setting                     | Recommended state                                  |
| --------------------------- | -------------------------------------------------- |
| Actions permissions         | Allow GitHub-authored and verified creator actions |
| Fork pull request workflows | Require approval for first-time contributors       |
| Workflow permissions        | `Read repository contents` by default              |
| Artifact/log retention      | `7` to `14` days                                   |

Use per-workflow `permissions:` blocks only when a workflow truly needs write access.

### Settings > Security

Recommended security state:

| Feature                         | Recommended state                                          |
| ------------------------------- | ---------------------------------------------------------- |
| Dependency graph                | On                                                         |
| Dependabot alerts               | On                                                         |
| Dependabot security updates     | On                                                         |
| Private vulnerability reporting | On                                                         |
| Secret scanning                 | On if visible for the repository                           |
| Code scanning                   | On if you want GitHub-managed analysis or later add CodeQL |

Private vulnerability reporting matters for a public repo because it gives outside researchers a private path that does not depend on them reading `SECURITY.md` first.

### Settings > Collaborators / Access

Keep write access small. For early public release, use:

- maintainer access only for active maintainers
- triage access for community helpers
- avoid unnecessary admin grants

## Wiki

Recommended initial state: **disabled**.

Reason: this repository already has versioned documentation in `docs/`, and running both a wiki and in-repo docs usually creates drift.

If you later enable the wiki, use this page map:

| Wiki page                    | Content                                            |
| ---------------------------- | -------------------------------------------------- |
| Home                         | What ArtBot is, supported surfaces, where to start |
| Installation                 | Node, pnpm, OpenAI-compatible model endpoints, Docker paths |
| Configuration                | `.env`, auth profiles, provider settings           |
| Source Adapters              | Supported sources, access modes, limitations       |
| Evidence and Outputs         | `var/runs/`, reports, screenshots, traces          |
| Operations                   | Local API, worker, logs, troubleshooting           |
| Development                  | Monorepo structure, tests, package map             |
| Security and Responsible Use | Terms, licensed sources, disclosure policy         |

## Releases and Tags

Use semantic tags in this form:

- `v0.2.0`
- `v0.2.1`
- `v0.3.0`

Release notes should include:

- what changed
- who it is for
- any migration or setup impact
- known limitations

Use GitHub's autogenerated release notes with `.github/release.yml`, then edit the draft before publishing.
